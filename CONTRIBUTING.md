# Contributing to RAGMap

感谢你考虑为 `RAGMap` 做贡献。

这个项目不是“资料堆积仓库”，而是一个强调方法脉络、研究导航和持续维护的知识仓库。因此，贡献的核心标准不是“内容多”，而是“结构清楚、信息密度高、能帮助读者形成地图感”。

## 我们欢迎什么样的贡献

- 修正文档中的事实错误、过时表述或不清晰表达
- 补充新的 RAG 方法，但必须说明其历史位置与问题意识
- 补充更好的方法对比、设计取舍分析和适用场景判断
- 增加高质量参考资料，包括论文、项目、博客、benchmark
- 增加图示、流程图、表格，让 README 和 docs 更适合 GitHub 展示
- 增加适合初学者理解的解释，但不要牺牲研究深度

## 提交内容时请尽量遵守这些原则

### 1. 不要只堆论文

如果你新增一个方法，请至少回答下面 4 个问题：

1. 它为什么出现？
2. 它想解决什么问题？
3. 它的主要代价和局限是什么？
4. 它和前一代方法相比，真正新增了什么？

### 2. 先讲脉络，再讲细节

我们更希望看到：

- “GraphRAG 为什么出现”
- “Memory RAG 相比 GraphRAG 换了什么思路”
- “Thinking-oriented RAG 为什么不是简单 rerank”

而不是直接进入模型名和指标。

### 3. 对 CognitiveRAG 保持客观

如果你补充 CognitiveRAG 相关内容，请保持以下写法：

- 放在整体 RAG 演进脉络中解释；
- 强调它试图解决的问题；
- 同时指出它带来的复杂度和待验证点；
- 避免宣传式语言。

### 4. 面向 GitHub 阅读体验

请优先考虑 Markdown 可读性：

- 标题层级清晰
- 段落不要过长
- 尽量使用表格、列表、导航链接
- 结论要能快速扫描

## 推荐的贡献方式

### 修正文档

- 直接提交 Pull Request
- 在 PR 描述中简单说明修改原因

### 新增方法章节或比较内容

建议在 PR 中说明：

- 新增内容属于哪条演进主线
- 它补的是哪一块空白
- 是否影响 README 导航结构

### 补充参考资料

请优先补充：

- 代表性论文
- 官方项目 / repo
- 高质量博客或讲解
- 公开 benchmark 或评测框架

并尽量避免：

- 低质量转载
- 缺少原始出处的二手总结

## 文档风格约定

- 以中文为主
- 写作风格偏研究导航，不是教科书，也不是营销文
- 尽量回答“为什么出现、解决什么、留下什么问题”
- 优先帮助读者形成方法地图
- 避免把章节写成纯时间线流水账

## 文件建议

- `README.md`：项目首页与总导航
- `docs/`：分主题章节
- `references.md`：参考资料与外部链接
- `glossary.md`：术语表
- `faq.md`：高频问题
- `ROADMAP.md`：后续维护计划

## Pull Request 建议模板

你可以在 PR 描述中简要回答：

1. 这次改了什么？
2. 为什么需要这次修改？
3. 它补足了仓库中的哪一块？
4. 是否需要同步更新 README 或相关阅读路线？
