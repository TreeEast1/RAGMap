# 01. 什么是 RAG

## 一句话理解

RAG（Retrieval-Augmented Generation，检索增强生成）可以理解为：

> 在让大模型回答问题之前，先去外部知识库里找证据，再基于这些证据生成答案。

它的核心意义不只是“多查一点资料”，而是把大模型从“只能依赖参数记忆”变成“可以动态接入外部知识”的系统。

## 为什么 RAG 会出现

如果只依赖模型参数，通常会遇到几个现实问题：

- 模型知识会过时
- 私有知识不在预训练语料中
- 某些问题需要明确证据来源
- 模型可能会编造听起来合理但不真实的答案

RAG 的出现，本质上是在补一条链路：

```text
用户问题
  -> 去外部知识源找信息
  -> 把找到的内容送给模型
  -> 让模型基于证据回答
```

因此，RAG 不是一个“模型替代品”，而是一个让模型与外部知识系统协同工作的框架。

## RAG 和“直接问模型”有什么不同

| 方式 | 知识来源 | 优点 | 问题 |
| --- | --- | --- | --- |
| 直接问 LLM | 模型参数内部 | 简单、快 | 容易过时，难溯源 |
| RAG | 参数知识 + 外部知识库 | 可更新、可扩展、可追踪 | 系统设计更复杂 |

所以 RAG 的关键不是“让模型更聪明”，而是“让模型回答时更有依据、更贴近当前知识”。

## RAG 的最小工作闭环

一个最基础的 RAG 系统通常包含以下步骤：

1. 收集文档
2. 切分文档为 chunk
3. 对 chunk 做 embedding
4. 建立向量索引
5. 用户提问时检索相关 chunk
6. 对检索结果做筛选或重排
7. 把问题和证据一起送给模型生成答案

这条链路很重要，因为后面几乎所有复杂 RAG，都是在这条链路上做增强。

## RAG 真正解决的是什么

从产品和系统角度看，RAG 主要在解决 4 类问题：

- 知识更新：不必重新训练模型也能接入新知识
- 私域知识利用：让模型回答企业文档、内部资料、个人知识库里的内容
- 证据增强：回答更容易附带依据
- 成本折中：相比全面微调，在很多任务上更灵活

## 但 RAG 不是万能的

很多人第一次接触 RAG，会以为只要“接一个向量库”就结束了。实际上不是。

基础 RAG 很快就会遇到这些问题：

- 召回到的 chunk 不一定完整
- 多个 chunk 之间的关系可能断裂
- 面对多跳问题时，单次检索不够
- 文档的全局结构往往被切碎了
- 即便检索到了证据，生成阶段仍然可能不稳定

也正因为这些问题，RAG 才会继续往 GraphRAG、MemoryRAG、Thinking RAG 等方向演化。

## 应该如何理解 RAG 的发展主线

可以用一个非常简单的视角来抓住主线：

- 第一阶段：先把相关内容找回来
- 第二阶段：发现“找回来”不等于“组织得好”
- 第三阶段：发现“组织得好”也不等于“推理得好”
- 第四阶段：开始把检索、记忆、推理、规划放进统一系统

这也是 `RAGMap` 试图建立的核心认知框架。
