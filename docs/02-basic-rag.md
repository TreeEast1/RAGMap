# 02. 基础 RAG

## 基础 RAG 的标准流程

所谓“基础 RAG”，通常指最常见的一类系统：

```text
文档 -> 切块 -> 向量化 -> 建索引 -> 检索 -> 重排 -> 生成
```

这条链路之所以重要，是因为后续几乎所有 RAG 变体，都是在其中某个环节发现瓶颈并做增强。

## 核心组件拆解

### 1. Chunking

把长文档拆成较小文本片段，便于索引和召回。

它看起来只是预处理，但实际上非常关键：

- chunk 太小：语义容易碎
- chunk 太大：召回容易不精准
- 固定切分：可能破坏自然结构
- 结构感知切分：实现更复杂，但往往效果更稳

### 2. Embedding

embedding 负责把文本变成向量，使语义相近的文本在向量空间中靠近。

这是基础 RAG 的检索基础，但它也带来一个隐含假设：

> “语义相近”足以代表“对回答有用”。

而这个假设，在复杂任务里并不总成立。

### 3. Retrieval

检索阶段根据用户问题，从向量库里找出最相关的候选 chunk。

常见做法包括：

- dense retrieval
- hybrid retrieval
- metadata filtering
- top-k recall

这个阶段决定了“候选证据池”的质量。

### 4. Reranking

第一阶段召回往往会包含噪声，因此很多系统会在召回后增加 reranking。

它的作用是：

- 提升最相关证据排位
- 降低弱相关内容干扰
- 为后续生成压缩上下文预算

但它通常仍然是“局部相关性排序”，未必能解决跨文档关系问题。

### 5. Generation

最后一步是把问题与召回证据一起送给 LLM 生成回答。

这一步的风险在于：

- 模型可能忽略部分证据
- 模型可能错误整合多个 chunk
- 模型可能在证据不足时补全想象

所以“检索到了”不等于“回答就稳了”。

## 基础 RAG 为什么流行

基础 RAG 之所以成为很多团队的起点，是因为它有三个现实优势：

- 工程路径清楚，搭建门槛低
- 与现有向量数据库生态适配好
- 对 FAQ、知识库问答、文档助手这类任务足够有效

如果任务相对简单，基础 RAG 往往已经能带来明显收益。

## 基础 RAG 的典型优点

| 维度 | 表现 |
| --- | --- |
| 上手难度 | 低 |
| 工程成熟度 | 高 |
| 可解释性 | 中等，至少能看到召回证据 |
| 知识更新能力 | 强 |
| 适合任务 | 文档问答、企业知识助手、FAQ |

## 基础 RAG 的核心问题

### 1. 召回不足

如果问题表达方式和文档表述差异较大，向量检索不一定能召回真正关键内容。

### 2. 上下文碎片化

当文档被切成 chunk 后，原本连续的逻辑链路被打断。模型看到的是一堆相关片段，而不是结构完整的知识。

### 3. 多跳推理弱

很多复杂问题并不是“找一个答案片段”，而是需要：

- 先找到 A
- 再根据 A 找到 B
- 最后综合 A 与 B

基础 RAG 对这种路径式问题支持较弱。

### 4. 全局关系感知不足

基础 RAG 更擅长局部匹配，不擅长全局知识组织。例如：

- 一个主题在多个文档中分散出现
- 实体之间存在跨段关系
- 用户问题本质上在问整体结构而不是局部事实

### 5. 幻觉与证据不稳定

即便检索到了相关证据，生成阶段仍可能出现：

- 证据拼接错误
- 引用不充分
- 过度补全

## 如何看待基础 RAG 的位置

基础 RAG 不是“落后方案”，而是整个 RAG 世界的基座。

后续很多新方向，并不是推翻它，而是在补它的短板：

- GraphRAG 在补“结构组织”；
- MemoryRAG 在补“记忆组织”；
- Thinking RAG 在补“检索与推理协同”。

如果不理解基础 RAG，后面很多方法会看起来像一堆术语；  
如果理解了基础 RAG 的局限，后面的演化就会顺得多。
