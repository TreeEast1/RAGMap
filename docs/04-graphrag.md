# 04. GraphRAG

## GraphRAG 为什么会出现

GraphRAG 的出现，不是为了“让 RAG 看起来更高级”，而是因为基础向量 RAG 在很多问题上确实不够。

尤其当任务涉及以下需求时，基础 RAG 的局部语义匹配会显得很吃力：

- 需要跨文档理解一个主题
- 需要建模实体之间的关系
- 需要处理多跳问答
- 需要先理解全局，再回答局部

GraphRAG 正是在这种背景下出现的：  
它试图把知识从“零散 chunk 集合”重新组织成“带关系的结构化图”。

## 为什么需要图结构

图结构的核心价值不在“形式上是图”，而在于它能表达三类重要信息：

### 1. 实体与关系

知识不再只是文本片段，而是：

- 实体
- 实体之间的关系
- 关系所在上下文

### 2. 局部与全局的桥梁

图允许系统：

- 从局部节点出发
- 沿关系扩展
- 最终形成对整个主题或社区的理解

### 3. 社区级聚合

图中的节点可以进一步聚类成社区，社区可以被摘要。  
这意味着系统不仅能回答“某个点”，还可能回答“这个主题整体上是什么样”。

## Microsoft GraphRAG 的核心思想

微软 GraphRAG 的一个核心贡献，是把知识组织过程显式化了。

可以粗略理解为：

1. 从文档中抽取实体与关系
2. 构建知识图
3. 对图进行社区划分
4. 为不同层级社区生成摘要
5. 回答时既可以从局部实体出发，也可以从社区摘要出发

这个思路的重要性在于：

- 它不再只依赖 chunk 相似度
- 它试图把“全局结构理解”放进 RAG

## GraphRAG 主要解决什么问题

| 问题 | GraphRAG 的解决思路 |
| --- | --- |
| 全局主题难以把握 | 通过社区摘要建立主题级表示 |
| 多跳关系弱 | 通过图中的路径和邻接关系支撑链式关联 |
| 文本被切碎 | 通过实体-关系重建部分知识结构 |
| 跨文档整合弱 | 图天生适合连接分散信息 |

## 它的价值不只是“更准”

GraphRAG 真正吸引研究者的，不只是指标提升，而是它改变了 RAG 的知识组织方式：

- 从片段库到结构网络
- 从局部召回到层级摘要
- 从相似性匹配到关系感知

这意味着它代表的是一种新范式，而不是仅仅一个工程技巧。

## 它引入了哪些新的成本和复杂度

GraphRAG 并不是“白拿收益”。它会显著引入几个成本：

### 1. 预处理成本高

要做实体抽取、关系抽取、图构建、社区发现、摘要生成。  
这比基础 RAG 的“切块 + embedding”复杂得多。

### 2. 对抽取质量敏感

如果实体抽错、关系抽错、社区划分不合理，后面整个图都可能偏掉。

### 3. 动态更新难

图结构系统通常不如向量索引那样容易增量更新。  
一旦知识库频繁变动，维护成本会明显上升。

### 4. 工程链路更长

从开发者视角看，GraphRAG 常常意味着：

- 更多中间模块
- 更多缓存与索引层
- 更多调试点
- 更复杂的线上稳定性问题

## 它适合哪些任务

GraphRAG 通常更适合：

- 多文档综合问答
- 主题级总结
- 复杂实体关系分析
- 需要全局视角的问题
- 高价值复杂知识库场景

## 它不太适合哪些任务

如果任务是：

- 简单 FAQ
- 单文档局部查询
- 对时延和成本极度敏感
- 知识结构本身较弱

那么 GraphRAG 不一定划算。

## 如何理解 GraphRAG 的历史位置

GraphRAG 的真正意义，不只在于它本身，而在于它把一个问题抬到了台面上：

> RAG 不能只做检索，还必须做知识组织。

后续很多方法，包括轻量化 GraphRAG、MemoryRAG、CognitiveRAG，都可以视为对这个问题的进一步展开。
