# 07. Reasoning-enhanced / Thinking-oriented RAG

## 为什么“只检索”还不够

即便系统已经具备：

- 更好的检索器
- 更好的结构化知识组织
- 更好的记忆表示

仍然会有一类问题很难：

> 问题本身需要被思考、分解、规划，而不是直接匹配。

这就是 Thinking-oriented RAG 出现的原因。

## 复杂问题的本质不是“答案难找”，而是“路径难确定”

很多问题并不是一个静态检索动作能解决的，例如：

- 一个问题包含多个隐含子问题
- 需要多轮检索逐步逼近答案
- 需要先判断应该走哪条知识路径
- 需要根据中间结果决定下一步检索方向

这时候，系统不能只问“最相似的文本是什么”，而必须问：

- 这个问题应该先拆成什么？
- 先检索哪一类证据？
- 哪些中间结果值得继续展开？
- 最终如何组合成回答？

## Think-on-RAG 一类思路的价值

这类方法的重要性在于，它把 retrieval 与 reasoning 不再看成两个孤立模块，而是看成协同过程。

粗略来说，流程会从：

```text
query -> retrieve -> answer
```

变成：

```text
query -> think -> decompose -> retrieve -> think -> retrieve -> synthesize
```

这意味着系统开始具备“动态路径”的概念。

## 这类方法通常怎么做

### 1. 查询分解

把一个复杂问题拆成多个子问题，分别检索。

### 2. 中间推理状态

系统不再一次性完成全部推理，而是在每一步保留中间结果，指导下一步检索。

### 3. 动态检索路径

不同问题可能会走不同检索链路，而不是统一 top-k。

### 4. 检索与推理循环

检索不是终点，而是推理过程中的一个反复调用模块。

## 它解决了什么

| 问题 | Thinking-oriented RAG 的改进 |
| --- | --- |
| 单次检索不够 | 允许多步检索 |
| 问题太复杂 | 先分解再求解 |
| 缺乏路径感 | 引入中间推理与决策 |
| 证据使用松散 | 让检索围绕推理过程展开 |

## 它带来的新代价

### 1. 时延更高

多步检索和多步推理通常意味着更高的响应时间。

### 2. 成本更高

更多 LLM 调用、更多中间状态维护、更多控制逻辑。

### 3. 稳定性要求更高

如果问题分解本身错了，后面整个路径都可能偏掉。

### 4. 系统设计更复杂

设计者不仅要会做检索，还要会做流程控制和推理编排。

## 与 MemoryRAG / GraphRAG 的关系

Thinking RAG 并不是在取代 GraphRAG 或 MemoryRAG。  
更准确地说，它们解决的问题不完全一样：

- GraphRAG 更关注知识结构
- MemoryRAG 更关注知识组织与记忆形态
- Thinking RAG 更关注问题求解过程

它们完全可能组合在一起。

## 为什么它是重要方向

因为很多真实复杂任务，最终都不只是“知识是否存在”，而是：

- 能否找到合适路径
- 能否逐步构造答案
- 能否让检索服务于推理，而不是只做前置过滤

这使 Thinking-oriented RAG 成为通往更强 agentic / planning 系统的重要桥梁。
