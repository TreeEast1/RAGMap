# 03. 为什么基础 RAG 不够

## 这不是“效果不好”那么简单

很多人第一次发现基础 RAG 有问题，会把它总结成一句很模糊的话：

> “召回不准，效果一般。”

但如果只停留在这句话，很难真正理解后续方法为什么会出现。

更准确地说，基础 RAG 的问题来自它的核心前提：

> 把知识切成局部 chunk，然后依赖语义相似度找回“相关片段”。

这个前提对简单任务有效，但对复杂任务会暴露系统性缺陷。

## 问题 1：局部匹配不等于全局理解

向量检索擅长找到“和问题语义相近”的片段，但很多问题需要的是：

- 整个主题的全局轮廓
- 跨多个文档的综合关系
- 从多个局部证据中拼出整体结构

基础 RAG 往往只能给出几块局部证据，却无法天然表达这些证据之间的结构关系。

## 问题 2：切块破坏了原生知识结构

文档原本可能有：

- 层级结构
- 实体关系
- 时间顺序
- 事件因果
- 社区主题

一旦把文档切成 chunk，很多结构信息就丢了。

于是系统变成：

- 能检索“像答案的片段”
- 但未必知道这些片段在整体知识网络中的位置

这正是 GraphRAG 类方法出现的重要背景。

## 问题 3：复杂问题不是一次检索就能解决

基础 RAG 通常假设：

```text
问题 -> 一次检索 -> 若干证据 -> 回答
```

但现实中的很多问题更像：

```text
问题
  -> 分解子问题
  -> 针对子问题做多次检索
  -> 在中间结果基础上继续搜索
  -> 汇总后再回答
```

这意味着很多复杂 RAG 系统的问题，不只是“检索库不够大”，而是“检索流程设计过于单步化”。

## 问题 4：相关性不等于有用性

向量检索关注的是相似性，而回答任务真正需要的是“有用证据”。

这两者并不总一致。

例如：

- 某个 chunk 与问题表面非常相似，但缺乏关键因果信息
- 某个 chunk 和问题字面不近，但恰好是推理链上最关键的桥节点

因此，后续很多方法开始从“相关检索”走向“任务导向检索”。

## 问题 5：证据融合阶段缺乏强约束

基础 RAG 把多个 chunk 丢给 LLM，希望模型自动融合。

但模型并不会天然做对这件事。常见问题包括：

- 选错重点
- 漏用关键证据
- 把来自不同来源的信息拼错
- 在证据不足时自行补全

这就是为什么后续方法开始考虑：

- 显式结构
- 中间记忆单元
- 查询分解
- 检索路径规划
- 认知式重排

## 基础 RAG 不够，催生了哪些新方向

| 不足 | 自然引出的方向 |
| --- | --- |
| 全局关系弱 | GraphRAG |
| 图构建太重 | 轻量化 GraphRAG |
| 知识只是片段集合 | Memory RAG |
| 单步检索不足 | Thinking / Reasoning RAG |
| 检索、记忆、重排割裂 | CognitiveRAG |

## 一个更重要的理解：RAG 的问题正在从“搜索问题”变成“知识组织问题”

早期很多人把 RAG 理解成：

- 文档切块
- 向量检索
- top-k 拼 prompt

但随着任务复杂度上升，RAG 的核心问题逐渐变化：

- 不只是“搜到什么”
- 更是“知识怎样被组织”
- 以及“模型怎样沿着合理路径使用这些知识”

这就是 RAG 从基础向量检索走向 Graph、Memory、Thinking、Cognitive 等方向的根本原因。
